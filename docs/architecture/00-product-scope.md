# Product Scope

## Problem

People need a fast, dependable place to write Markdown without losing work. The editor must feel immediate during writing, preserve documents offline, render Markdown predictably, and optionally keep a user's documents available on other devices.

## Primary users

| User | Primary job | Success condition |
| --- | --- | --- |
| Individual writer | Draft notes, documentation, and articles. | Can create, organize, preview, and export without data loss. |
| Technical user | Edit repository-like Markdown with code blocks and links. | Formatting and source remain exact and portable. |
| Returning user | Continue work from another browser or device. | Recent changes reconcile safely and are discoverable. |

## Functional requirements

1. Create, rename, move, archive, restore, and permanently delete documents.
2. Organize documents in workspaces and hierarchical folders.
3. Edit standard CommonMark plus GitHub Flavored Markdown (GFM) extensions.
4. Provide source, split, and preview modes; preserve the selected mode per user.
5. Autosave locally on each meaningful content change and sync when online.
6. Search titles and document bodies within the active workspace.
7. Support import/export of `.md`; export rendered HTML and PDF in a later milestone.
8. Support guest/local-only use, then account creation without discarding local work.
9. Record recoverable document revisions and expose conflict resolution when concurrent edits occur.

## Non-functional requirements

- Typing must not wait for network I/O or a preview render.
- A content edit must survive a tab crash after the local persistence debounce completes.
- Preview output must not execute untrusted scripts or unsafe URLs.
- The standard editing path must be usable with keyboard only and meet WCAG 2.2 AA.
- The API must enforce tenant boundaries independently of client state.

## Explicit exclusions for the first release

- Simultaneous multi-cursor collaboration and shared cursors.
- Arbitrary HTML execution, embedded scripts, and third-party widgets.
- Full repository management, Git synchronization, and a desktop filesystem watcher.
- Rich-text as the primary editing model; Markdown source remains canonical.
- End-to-end encryption. This changes search, recovery, and support operations and requires a separate design.

## Product metrics

| Metric | Initial target |
| --- | --- |
| Local save success | 99.99% of attempted local writes |
| Cloud sync median latency | Under 2 seconds after connectivity returns |
| Lost confirmed edits | Zero |
| Preview time for a 100 KB document | Under 300 ms on a supported desktop |
| Core Web Vitals | LCP under 2.5 s on the authenticated app shell |

## Terminology

- **Workspace**: tenant boundary containing folders and documents.
- **Document**: Markdown source plus metadata, revisions, and lifecycle state.
- **Revision**: immutable snapshot generated by a successful server write.
- **Operation**: a queued local intent, such as update, move, or delete.
- **Conflict**: a rejected update because its expected document version is no longer current.
